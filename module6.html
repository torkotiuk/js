// Decision 1 - good to understand & enough for work

const stat = allTags.reduce((acc, element) => {
// check if property exists, if exists, add to value +1
if (acc[element]) {
acc[element] += 1;

return acc;
}

acc[element] = 1;

return acc;

// Decision 2 - linter doesn't like mutable code, like "acc[element] += 1;", so make unmutable code

const stat = allTags.reduce((acc, element) => {
// check if property exists, if exists, add to value +1
if (acc[element]) {
// we didn't mutate acc, we create new obj acc
return {
// firstly spread "acc"
...acc,

// then to exists property "[element]" (html f.e.) add + 1
[element]: acc[element] + 1
};
}

// if we don't have necessary property (like "html"), we didn't mutate acc, we spread acc & add new property
return {
...acc,
[element]: 1
};
}, {});

sort mutable

Array.prototype.sort(callback(currentEl, nextEl){
//
})

- sort and mutate an original array;
- by default:
☑ sorts an array alphabetically (sorts symbols in ascending order);
☑ convert value to string & sort by "unicode character table" (symbl.cc)

Mutable code

F. e.

const num = [1, 8, 2, 12, 10];

console.log(num.sort()); // [1, 10, 12, 2, 8]
// b/c alphabetically string "12" is less than "2" etc.

F. e.

const letters = ['C', 'B', 'a', 'A', 'b'];;

console.log(letters.sort()); // ['A', 'B', 'C', 'a', 'b']

In frontend mostly sorts objects in arrays, so we should use compareFunction(currentElement, nextElement) (callback) to sort not only alphabetically.

compareFunction(curEl, nextEl) rules:

  • curEl - nextEl = positive/negative - simplest alphabetically sort, like without callback

    f.e. with num [1, 8, 2, 12, 10]

    element1: 1 - 8 = -7 -> negative value, so nothing to change;
    [1, 8, ... ]
    element2: 8 - 2 = 6 -> positive value, so elements changes places;
    [..., 2, 8, ... ]
    element3: 8 - 12 = -4 -> negative value, so nothing to change;
    [..., 8, 12, ... ]

  • nextEl - curEl = positive/negative - sorts numbers in descending order

  • nextEl - curEl = 0 - nothing to sort

Create immutable code

const numbers = [1, 8, 2, 12, 10];

const numbersCopy = [...numbers];

console.log(numbers === numbersCopy); // false

Or create array copy and sort it at the same time:

const numbers = [1, 8, 2, 12, 10];

const descendingNumbers = [...numbers].sort((a, b) => b-a );

console.log(descendingNumbers); // [12, 10, 8, 2, 1]

get() ( _.get() ) _.get(object, path, [defaultValue])

F.e. - when from backend comes object & we have to use "city" property:

A) object is OK:

const user = {
name: "Mango",
location: {
coords: [1, 2],
city: London }
}

Without any library:

console.log(user.location.city); // London

With lodash library the same:

console.log(_.get(user, "user.location.city")); // London

There is no difference with code if everything ok, but sometimes backend return not full object.

B) Backend return not full object:

const user = {
name: "Mango",
}

// With lodash library the same:

console.log(_.get(user, "user.location.city")); // undefined

// Without any library:

console.log(user.location.city); // error: Cannot read properties of undefined (reading 'city')

To sum up, lodash get() method return "undefined" from any deep object property when property doesn't exists, & we will have still working code.

C) object isn't full, JS decision:

Lodash made decision (with no full object properties ) long time ago, in JS this problem solved only couple years ago by operator "?":

const user = {
name: "Mango",
}

// Without any library:

console.log(user?.location?.city); // error: Cannot read properties of undefined (reading 'city')
